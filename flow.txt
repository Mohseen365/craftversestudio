# Bouquet Project – Comprehensive Mindmap & Flow

## 1. Core Objective
Solve the "Capacity Constraint" problem: Can we finish a new bouquet order before its deadline without making existing orders late?

---

## 2. Global Architecture
- **Framework:** Next.js (Server-First approach).
- **ORM:** Prisma (Optimized with precise `select` trees).
- **Database:** Supabase/PostgreSQL (Strict 1-connection limit on free tier).
- **Rendering:** All data-heavy pages marked `force-dynamic` to protect build-time stability.
- **State Management:** Localized to components (`OrderCard`, `ProductItem`) to prevent mass re-renders.

---

## 3. Customer Flow (Who Calls What)

### [A] Discovery & Product View
- **Home (`/`) / Catalog (`/catalog`):**
  - Calls `prisma.product.findMany` (Server-side).
  - Renders `ProductCard`.
- **Product Page (`/products/[slug]`):**
  - Calls `prisma.product.findUnique`.
  - Tracks `PRODUCT_VIEW` (Fire-and-forget).

### [B] Ordering & Identity
- **Order Page (`/order/[slug]`):**
  - Renders `OrderForm`.
- **Order Submission (`POST /api/orders`):**
  - Creates Guest User if no session (`/api/customer/guest`).
  - Creates `Order` (status: `PENDING_REVIEW`).
  - Creates `OrderItem` & `Address`.
  - Redirects to Login.
- **Login/Contact Details (`/login`):**
  - Customer provides Name/Mobile.
  - Calls `POST /api/customer/login`.
  - Merges guest order to the "permanent" user record.

### [C] Payment & Tracking
- **Payment Page (`/order/[slug]/payment`):**
  - **NEW:** Server-side verification of `orderId`.
  - Renders `PaymentUpload`.
- **Upload (`POST /api/upload` -> `POST /api/orders/[id]/payment`):**
  - Saves screenshot URL.
  - Moves Order status to `PAYMENT_VERIFICATION`.
- **Track Order (`/track`):**
  - **NEW:** Server-side lookup if query params present.
  - Calls `GET /api/orders/track`.
  - Displays `OrderStatusBadge`.

## 4. Admin Flow (What Happens Where)

### [A] Order Management (`/admin/orders`)
- **Reviewing:** Admin enters `shippingDurationDays`.
- **Capacity Preview:** Keystroke triggers `GET /api/admin/capacity?productionDeadline=...`.
  - Calls `checkAcceptability` (In-memory algorithm run).
- **Acceptance:** Admin clicks "Accept Order".
  - Calls `POST /api/orders/[id]/accept`.
  - Updates status to `ACCEPTED`.
  - Triggers `persistAllocations` (Writes to `CapacityReservation` table).

### [B] Production Workflow
- **Status Progression:** Admin moves order through buttons:
  - `CONFIRMED` -> `IN_PRODUCTION` -> `READY_TO_SHIP` -> `SHIPPED`.
  - Calls `PATCH /api/admin/orders/[id]`.
  - Only triggers `rebuildSchedule()` if status leaves schedulable set (e.g., `SHIPPED`).

### [C] Capacity Planning (`/admin/capacity`)
- **Dashboard:** Server-side fetch of all capacity rows.
- **Manual Overrides:** Admin changes daily limit (e.g., Holidays = 0).
  - Calls `POST /api/admin/capacity/override`.
- **Progress Tracking (Mid-Day Reallocation):**
  - Admin records work done (e.g., "Order 1: 0.5 units done today").
  - Calls `POST /api/admin/capacity/progress`.
  - **Logic:** This locks 0.5 units. If the day's capacity is 1.0, the remaining 0.5 becomes "Available" for other orders.
  - Subsequent `rebuildSchedule()` calls respect this lock and reallocate the remaining 0.5 to higher-priority or urgent orders.

## 5. Scheduling Engine Logic (Deep Dive)

### [Step 1] Data Loading (`getSchedulerData`)
- Fetches all `SCHEDULABLE` orders.
- Optimized: Only selects `id`, `items(quantity, productionDays)`, `reservations(completedQuantity)`.

### [Step 2] In-Memory Algorithm (`scheduleOrdersInMemory`)
- Calculates `remainingEffort` (Required - Locked).
- Identifies viable date window (Today -> Deadline).
- Calculates `slack` (Window Capacity - Remaining Need).
- **Sorting:** Most constrained orders first (Fewest dates -> Lowest slack -> Earliest deadline -> Unique ID tie-breaker).
- **Allocation:** Deterministic greedy forward-filling of daily capacity.

### [Step 3] Persistence (`persistAllocations`)
- Deletes old `isManual: false` reservations.
- Performs `createMany` with flat scalar IDs.

## 6. Detected Inconsistencies & Improvement Suggestions

### [!] Inconsistency: User Identity Security
- **Current:** `bouquet_customer` cookie stores raw, unsigned User ID.
- **Risk:** Any user can change their cookie value to any other ID and "steal" orders.
- **Suggestion:** Use signed cookies or a proper JWT-based session in `lib/auth.ts`.

### [!] Inconsistency: Payment Submission Ownership
- **Current:** `POST /api/orders/[id]/payment` doesn't check if the current user owns the order.
- **Suggestion:** Cross-reference `order.userId` with `sessionUser.id` before allowing a screenshot upload.

### [!] Inconsistency: `ProductionProgress` Model
- **Current:** Exists in schema but never read or written to.
- **Suggestion:** Remove from `schema.prisma` to keep the DB lean.

### [!] Logic Suggestion: Waitlist Automation
- **Current:** Waitlist exists as a status but is manually managed.
- **Suggestion:** If `checkAcceptability` fails, offer a "Waitlist Me" button to the customer instead of just a rejection to the admin.

### [!] Performance Suggestion: DB Connection Pooler
- **Current:** 1 direct connection limit.
- **Status:** Already optimized queries to the bone.
- **Suggestion:** If scaling, use Supabase's transaction pooler (port 6543) instead of direct (port 5432) to handle more concurrent light queries.

## 7. Logic Validation Scenarios (Verified)

### [Scenario 1] Mid-Day Progress (Locking)
- **State:** Order A needs 2.0 units. Daily limit is 1.0.
- **Action:** Admin records 0.5 units done today.
- **Result:** Scheduler locks those 0.5 units for Order A. Order A's remaining 1.5 units are pushed to future dates if necessary. Today's "Planned" total for Order A is exactly 0.5.

### [Scenario 2] Urgent Order (Stealing Remaining Capacity)
- **State:** Order A has 0.5 units locked today. Limit is 1.0. Remaining today = 0.5.
- **Action:** Urgent Order B arrives (needs 0.5, deadline TODAY).
- **Result:** Order B is allocated the remaining 0.5 units today. Order A is pushed to the next available slot. Both meet their deadlines.

### [Scenario 3] Capacity Override (Staff Shortage)
- **State:** Limit is 1.0.
- **Action:** Admin overrides limit for tomorrow to 0.2.
- **Result:** Scheduler only allocates 0.2 units across all orders for tomorrow, pushing the overflow to the following days.

### [Scenario 4] Constraint Failure
- **State:** Order A has 0.5 locked today. Remaining today = 0.5.
- **Action:** Urgent Order C arrives (needs 0.8, deadline TODAY).
- **Result:** Scheduler rejects Order C with "Missing 0.30 units" reason. System integrity is maintained; Order A is not jeopardized.
