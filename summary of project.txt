# Bouquet Project – Technical Architecture & Business Flow (Complete Unified Document)

## System Identity (For AI Agents & Search)
```yaml
Project: Bouquet
Type: Capacity-aware production planning & order management platform
Stack: Next.js, Prisma, Supabase/PostgreSQL
Tier: Strict 1-connection limit (Free Tier)
Rendering: force-dynamic (all data-heavy pages)
Core Problem: "Can a new order be completed before its deadline without making existing orders late?"
Scheduling: Greedy, constrained, deterministic, most-constrained-first
Key Entities: Capacity, CapacityReservation (plannedQuantity, completedQuantity), Order, Product.productionDays
```

## Project Objective
A bouquet order management and production planning system that ensures orders are only accepted when sufficient production capacity exists between the order request date and production deadline.

The system combines:
* Product catalog management
* Customer order management
* Payment verification
* Production capacity planning
* Automatic scheduling
* Delivery tracking

The core problem being solved is:
> "Given limited production effort per day, can a new order be completed before its production deadline without causing existing accepted orders to miss theirs?"

# System Overview & Flow Map

## 1. Core Objective
Solve the "Capacity Constraint" problem: Can we finish a new bouquet order before its deadline without making existing orders late?

## 2. Global Architecture
- **Framework:** Next.js (Server-First approach).
- **ORM:** Prisma (Optimized with precise `select` trees).
- **Database:** Supabase/PostgreSQL (Strict 1-connection limit on free tier).
- **Rendering:** All data-heavy pages marked `force-dynamic` to protect build-time stability.
- **State Management:** Localized to components (`OrderCard`, `ProductItem`) to prevent mass re-renders.

## 3. Customer Flow (Who Calls What)

### [A] Discovery & Product View
- **Home (`/`) / Catalog (`/catalog`):**
  - Calls `prisma.product.findMany` (Server-side).
  - Renders `ProductCard`.
- **Product Page (`/products/[slug]`):**
  - Calls `prisma.product.findUnique`.
  - Tracks `PRODUCT_VIEW` (Fire-and-forget).

### [B] Ordering & Identity
- **Order Page (`/order/[slug]`):**
  - Renders `OrderForm`.
- **Order Submission (`POST /api/orders`):**
  - Creates Guest User if no session (`/api/customer/guest`).
  - Creates `Order` (status: `PENDING_REVIEW`).
  - Creates `OrderItem` & `Address`.
  - Redirects to Login.
- **Login/Contact Details (`/login`):**
  - Customer provides Name/Mobile.
  - Calls `POST /api/customer/login`.
  - Merges guest order to the "permanent" user record.

### [C] Payment & Tracking
- **Payment Page (`/order/[slug]/payment`):**
  - Server-side verification of `orderId`.
  - Renders `PaymentUpload`.
- **Upload (`POST /api/upload` -> `POST /api/orders/[id]/payment`):**
  - Saves screenshot URL.
  - Moves Order status to `PAYMENT_VERIFICATION`.
- **Track Order (`/track`):**
  - Server-side lookup if query params present.
  - Calls `GET /api/orders/track`.
  - Displays `OrderStatusBadge`.

## 4. Admin Flow (What Happens Where)

### [A] Order Management (`/admin/orders`)
- **Reviewing:** Admin enters `shippingDurationDays`.
- **Capacity Preview:** Keystroke triggers `GET /api/admin/capacity?productionDeadline=...` (debounced 400ms).
  - Calls `checkAcceptability` (In-memory algorithm run).
- **Acceptance:** Admin clicks "Accept Order".
  - Calls `POST /api/orders/[id]/accept`.
  - Updates status to `ACCEPTED`.
  - Triggers `persistAllocations` (Writes to `CapacityReservation` table).

### [B] Production Workflow
- **Status Progression:** Admin moves order through buttons:
  - `CONFIRMED` -> `IN_PRODUCTION` -> `READY_TO_SHIP` -> `SHIPPED`.
  - Calls `PATCH /api/admin/orders/[id]`.
  - Only triggers `rebuildSchedule()` if status leaves schedulable set (e.g., `SHIPPED`).

### [C] Capacity Planning (`/admin/capacity`)
- **Dashboard:** Server-side fetch of all capacity rows.
- **Manual Overrides:** Admin changes daily limit (e.g., Holidays = 0).
  - Calls `POST /api/admin/capacity/override`.
- **Progress Tracking (Mid-Day Reallocation):**
  - Admin records work done (e.g., "Order 1: 0.5 units done today").
  - Calls `POST /api/admin/capacity/progress`.
  - **Logic:** This locks 0.5 units. If the day's capacity is 1.0, the remaining 0.5 becomes "Available" for other orders.
  - Subsequent `rebuildSchedule()` calls respect this lock and reallocate the remaining 0.5 to higher-priority or urgent orders.

## 5. Scheduling Engine Logic (Deep Dive)

### [Step 1] Data Loading (`getSchedulerData`)
- Fetches all `SCHEDULABLE` orders.
- Optimized: Only selects `id`, `items(quantity, productionDays)`, `reservations(completedQuantity)`.

### [Step 2] In-Memory Algorithm (`scheduleOrdersInMemory`)
- Calculates `remainingEffort` (Required - Locked).
- Identifies viable date window (Today -> Deadline).
- Calculates `slack` (Window Capacity - Remaining Need).
- **Sorting:** Most constrained orders first (Fewest dates -> Lowest slack -> Earliest deadline).
- **Allocation:** Greedy forward-filling of daily capacity.

### [Step 3] Persistence (`persistAllocations`)
- Deletes old `isManual: false` reservations.
- Performs `createMany` with flat scalar IDs.

# Core Concepts

## Capacity Unit
Capacity is measured in **production effort days**, not bouquet quantity. This is a foundational concept for the scheduler. A "production day" represents the work one unit of capacity can perform in a full day. An order's total cost is calculated by multiplying this per-product cost by the quantity ordered.

```text
1   = one full production day
0.5 = half production day
2   = two production days available
```

**Examples:**
* Order X: `productionDays = 1`, `quantity = 2` → `requiredCapacity = 2`. If daily capacity is `1`, this order requires two full production days.
* Order Y: `productionDays = 0.5`, `quantity = 4` → `requiredCapacity = 2`. This order consumes two production days even though four bouquets are ordered.

## Daily Capacity
Each date has a configurable production capacity stored in the `Capacity` table (field: `maximumCapacity`).

Default capacity constant: `DAILY_PRODUCTION_CAPACITY = 1` (defined in `lib/constants.ts`).

Per-date overrides are stored in the `Capacity` table and take precedence over the constant. Capacity can be manually adjusted from `CapacityPanel` via `POST /api/admin/capacity/override`.

**Capacity Override Examples:**
Overrides are used to represent real-world capacity changes that are not related to order work.
* **Holiday**: capacity = 0
* **Half Day**: capacity = 0.5
* **Extra Staff**: capacity = 2

> **Legacy Note:** The `Setting` model also stores a `DAILY_PRODUCTION_CAPACITY` key used by `getDailyProductionCapacity()` in `lib/capacity.ts`. However, this is a legacy helper; the active scheduling engine uses the `DAILY_PRODUCTION_CAPACITY` constant from `lib/constants.ts` as a fallback.

## Product.productionDays
Stored as `Decimal(10,2)` on the `Product` model. Represents the effort required to produce one unit of a product.

Example:
| Product         | productionDays |
| --------------- | -------------- |
| Rose Bouquet    | 1              |
| Premium Bouquet | 2              |
| Wedding Bouquet | 3              |

## Order Required Capacity
Order effort is calculated from order items at multiple points in the codebase.

**Formula:** `requiredCapacity = Σ (orderItem.quantity × product.productionDays)`

Used in:
* `api/orders/[id]/accept/route.ts` — before accepting
* `api/admin/capacity/route.ts` — for capacity preview in `OrderPanel`
* `lib/scheduler.ts → getSchedulerData()` — when building the schedule

## Production Deadline
The production deadline is the latest date by which all production work for an order must be completed to ensure timely delivery.

```text
occasionDate
    ↑
shippingDate  = occasionDate - shippingDurationDays
    ↑
productionDeadline = shippingDate - 1 day
```

Calculated in `lib/capacity.ts`:
* `calculateShippingDate(occasionDate, shippingDurationDays)`
* `calculateProductionDeadline(shippingDate)` — subtracts 1 day

Set on the Order when admin accepts it (via `/api/orders/[id]/accept`).

## Work Lifecycle
This describes the three-part model for tracking order effort over time.

* **plannedQuantity**: Future work reservation created by the scheduling engine. This represents an intention to do work and is not permanent.
* **completedQuantity**: Work already performed and recorded by an admin via `POST /api/admin/capacity/progress`. This work is **locked** and will never be modified by the scheduler.
* **remainingWorkForOrder**: `requiredCapacity - totalCompletedQuantity` (sum of all `completedQuantity` across all dates for an order).

Only remaining work participates in future scheduling. Completed work is never moved by `rebuildSchedule()`.

### Production Progress (Recording Work)
At the end of a production day, admins record how much effort was completed per order via `POST /api/admin/capacity/progress`.
* Recorded effort is capped at the `plannedQuantity` for that date.
* Once recorded, this effort becomes locked. Future rebuilds only consider unfinished effort.

### Mid-Day Reallocation of Capacity
Admins can manually adjust progress to react to urgent requests, effectively redistributing uncompleted work.
1. Admin reduces completed effort for a lower-priority order.
2. Remaining available effort for the day increases.
3. The remaining capacity can be assigned to an urgent order.
4. `rebuildSchedule()` recalculates future allocations using the updated totals.

# Order Status Flow

## Schedulable vs. Inactive Statuses
Only orders in `SCHEDULABLE_STATUSES` participate in scheduling. A key optimization is that `rebuildSchedule()` only runs when an order enters or leaves this set.

* **SCHEDULABLE_STATUSES**: `ACCEPTED`, `PAYMENT_PENDING`, `PAYMENT_SUBMITTED`, `PAYMENT_VERIFICATION`, `CONFIRMED`, `IN_PRODUCTION`, `READY_TO_SHIP`
* **INACTIVE_STATUSES** (excluded from scheduling): `CANCELLED`, `REFUNDED`, `REJECTED`, `SHIPPED`, `DELIVERED`, `PAYMENT_REJECTED`

Both defined in `lib/constants.ts`.

## Status Transitions
```
PENDING_REVIEW
  → ACCEPTED           (admin accepts via OrderPanel)
  → REJECTED           (admin rejects)

ACCEPTED
  → PAYMENT_PENDING    (after acceptance, customer needs to pay)

PAYMENT_PENDING / PAYMENT_SUBMITTED
  → PAYMENT_VERIFICATION  (customer submits payment screenshot)

PAYMENT_VERIFICATION
  → CONFIRMED          (admin verifies payment)
  → PAYMENT_REJECTED   (admin rejects payment)

CONFIRMED → IN_PRODUCTION → READY_TO_SHIP → SHIPPED → DELIVERED

Any non-terminal → CANCELLED / REFUNDED
```

# Scheduling Engine

Location: `lib/scheduler.ts`

Main exported functions:
* `scheduleOrdersInMemory()`
* `getSchedulerData()`
* `checkAcceptability()`
* `rebuildSchedule()`
* `persistAllocations()`
* `getSchedulerPlanningRows()`

## Dynamic Rescheduling (Urgent Order Philosophy)
Future reservations are not permanent. The scheduler follows a "most constrained first" approach to fairly allocate capacity. When a new order arrives:
1. Completed work remains locked.
2. Future reservations may be reassigned to make room.
3. Orders with earlier production deadlines receive priority.
4. Orders with later deadlines may be pushed into later available dates.

## Scheduler Invariants
The scheduler must always guarantee:
1. No date exceeds `maximumCapacity`.
2. Completed work is never modified.
3. An order cannot be scheduled after its `productionDeadline`.
4. `Remaining effort` = `requiredCapacity - totalCompletedQuantity`.
5. Fractional allocations are allowed.
6. Scheduling results must be deterministic.

## Sorting Strategy

*Reason: Orders with the least scheduling flexibility are allocated first. This prevents flexible long-deadline orders from consuming capacity needed by urgent orders.*

## scheduleOrdersInMemory(todayKey, orders, capacityLimits, completedWorkMap)

Pure in-memory scheduling algorithm. No DB access.

Parameters:
* `todayKey`: YYYY-MM-DD string for today
* `orders`: `OrderScheduleInput[]` — each has `id`, `orderNumber`, `productionDeadlineKey`, `requiredCapacity`, `lockedCapacity`, `status`
* `capacityLimits`: `Map<dateKey, maxCapacity>` — per-date limits from DB
* `completedWorkMap`: `Map<dateKey, completedWork>` — already-done work per date

Algorithm:
1. For each order, compute `remaining = requiredCapacity - lockedCapacity`
2. Build list of viable dates from `todayKey` to `productionDeadlineKey`
3. Compute `slack = totalWindowCapacity - remaining`
4. Sort orders: most constrained first (fewest dates → lowest slack → earliest deadline)
5. Walk forward through dates, allocate `min(need, available)` per day
6. If any order can't be fully allocated before its deadline → return `success: false` with reason

Returns: `{ success, allocations: Map<orderId, ScheduledAllocation[]>, reason? }`

---

## getSchedulerData(todayKey)

Pulls current scheduling context from DB.

Steps:
1. Fetch all orders with `SCHEDULABLE_STATUSES` that have a `productionDeadline`
2. Find the farthest `productionDeadline` among non-inactive orders (determines window end)
3. Fetch `Capacity` records in `[today, endDate]` window → build `capacityLimits` map
4. Fetch all `CapacityReservation` records in the window → build `completedWorkMap` (sum of `completedQuantity` per date)
5. Map orders to `OrderScheduleInput` — `lockedCapacity` = sum of `completedQuantity` from reservations

Returns: `{ inputs, capacityLimits, completedWorkMap }`

---

## checkAcceptability(candidate)

Used before accepting an order to validate capacity. This is the authoritative gate.

Parameters: `{ id, orderNumber, productionDeadline: Date, requiredCapacity }`

Process:
1. Calls `getSchedulerData(todayKey)` to get current state (can accept preloaded `SchedulerData` to skip duplicate DB fetch)
2. Removes the candidate order from inputs (to re-add it fresh)
3. Adds candidate with `lockedCapacity: 0` and `status: "ACCEPTED"`
4. Runs `scheduleOrdersInMemory()` with the updated inputs
5. If success → extracts allocation for candidate → returns `suggestedDates`

Returns:
```typescript
{ canAccept: boolean, reason?: string, suggestedDates: Array<{ date: string, quantity: number }>, allocations?, schedulerData? }
```

---

## rebuildSchedule()

Rebuilds ALL future capacity reservations from scratch. This function is now only called when a status change moves an order into or out of `SCHEDULABLE_STATUSES`.

Process:
1. Calls `getSchedulerData(todayKey)` (can accept preloaded `SchedulerData`)
2. Runs `scheduleOrdersInMemory()`
3. Upserts `Capacity` rows for every allocation date (outside transaction, required before FK inserts)
4. Builds a `Map<dateKey, capacityId>` from DB
5. In a DB transaction:
   - Deletes all `isManual: false` reservations with `date >= today`
   - Creates new `CapacityReservation` records using flat `capacityId` scalar (not nested connect — `createMany` does not support relations)
6. Calls `getSchedulerPlanningRows()` and returns rows

Throws if `scheduleOrdersInMemory()` returns `success: false`.

---

## persistAllocations(allocations, todayKey)

The write-only side of a rebuild. Takes a pre-computed allocation map from `scheduleOrdersInMemory()` and persists it. No DB reads, no algorithm run. This allows the accept flow to reuse a single scheduling run for both validation and persistence.

---

## getSchedulerPlanningRows()

Builds the planning table for the admin capacity UI.

Process:
1. Finds the farthest `occasionDate` among non-inactive orders (window end)
2. Fetches `Capacity` rows from today to window end, with full reservation joins (order → user → addresses, items → product)
3. Walks each date in the window day-by-day
4. For dates with a `Capacity` record: computes `used` (sum of `plannedQuantity`), `remaining`, `isFull`
5. Skips dates with no `Capacity` record

Returns array of rows (used by `GET /api/admin/capacity` when no query params).

# Capacity Reservation Model

```
CapacityReservation
  capacityId        → Capacity (date)
  orderId           → Order
  plannedQuantity   → effort allocated for future work (Decimal)
  completedQuantity → effort already done (Decimal, default 0)
  manualQuantity    → manual override amount (Decimal, default 0)
  isManual          → true for manual overrides (not deleted on rebuild)
```

Unique constraint: `(capacityId, orderId)` — one reservation per order per date.

`rebuildSchedule()` only deletes `isManual: false` future reservations, preserving manual ones.

## API: PATCH /api/admin/orders/[id]

Body schema (zod):
```typescript
{
  status?: OrderStatus,
  trackingNumber?: string,
  verifyPayment?: boolean,
  rejectPayment?: boolean
}
```

Logic:
1. If `verifyPayment: true`:
   - Sets all `UPLOADED` payments → `VERIFIED` (with `verifiedAt`)
   - Sets order status → `CONFIRMED`
   - **No rebuildSchedule()** — Both statuses are in `SCHEDULABLE_STATUSES`.
2. If `rejectPayment: true`:
   - Sets all payments → `REJECTED`
   - Sets order status → `PAYMENT_REJECTED`
   - Calls `rebuildSchedule()` — `PAYMENT_REJECTED` is `INACTIVE`, order leaves the schedule.
3. Generic status / trackingNumber update:
   - Updates order in DB
   - Calls `rebuildSchedule()` only if `schedulingAffected(fromStatus, toStatus)` is true — i.e. the transition crosses the `SCHEDULABLE` ↔ `INACTIVE` boundary.

## API: POST /api/orders/[id]/accept

Called by OrderPanel's "Accept Order" button.

Process:
1. Requires admin auth
2. Parses `{ shippingDurationDays }` from body (validated as `z.number().min(0).max(365)`)
3. Validates order is in `PENDING_REVIEW` status
4. Fetches order with items + productionDays
5. Computes `shippingDate` and `productionDeadline`
6. Computes `requiredCapacity = Σ(item.quantity × productionDays)`
7. Calls `checkAcceptability(...)` (preloads `schedulerData` to avoid a duplicate DB fetch)
8. If `!check.canAccept` → returns 409 with reason
9. Updates order:
   - status → `ACCEPTED`
   - `shippingDurationDays`, `shippingDate`, `productionDeadline`
   - Resets all payments to status `PENDING`
10. Calls `persistAllocations(check.allocations, ...)` to save the schedule without re-running it
11. Returns `{ success: true, suggestedDates }`

# Scheduler Logic Validation (Mathematically Proven)

The scheduling engine follows a strict **Greedy Constrained Allocation** model.

### Key Logic Scenarios Verified:
1. **Mid-Day Progress (Locking)**: Recorded work (`completedQuantity`) is subtracted from `requiredCapacity`. The remaining effort is then rescheduled. Existing progress is **never** moved.
2. **Urgent Orders**: When an urgent order arrives with a tight deadline, the scheduler assigns it the remaining available capacity for that day. Any lower-priority, more flexible orders are pushed into later available slots.
3. **Staff Shortages**: Fractional capacity overrides (e.g., 0.2 units) are strictly respected. Overflow work is automatically deferred to the first available date with surplus capacity.
4. **Holidays**: Zero-capacity days (maximumCapacity = 0) correctly act as barriers, with no work being scheduled on those dates.
5. **Determinism**: Ties in constraints (same deadline, same available window) are broken by unique Order IDs, ensuring the production plan is consistent and reproducible across every rebuild.

# Key Fixes & Optimizations Log

## Scheduling & Data Integrity Fixes
* **CRITICAL Fix:** `rebuildSchedule()` used `createMany` with nested relation `connect`, which is not supported by Prisma. This silently stripped the FK, causing insert failures. Fixed by upserting `Capacity` rows first and passing flat `capacityId` scalars.
* **Double-Algorithm Run:** The accept flow called `getSchedulerData` and `scheduleOrdersInMemory` twice. Optimized by splitting `persistAllocations()` from `rebuildSchedule()` and preloading `SchedulerData` into `checkAcceptability()`, resulting in a single DB fetch and algorithm run per acceptance.
* **Payment Verification Bug:** `verifyPayment` filtered payments by status `PENDING`, but payments are set to `UPLOADED` on upload. Fixed filter to check for `UPLOADED`.
* **Timezone Bug:** Progress updates used local timezone `toDateOnly()`, which mismatched UTC-stored `Capacity.date`. Fixed by using UTC-only `formatDateKey` and `parseDateKey`.
* **Capacity Panel Data:** Expanded view showed `plannedQuantity` for all columns (Total, Allocated, Completed). Fixed by adding `totalRequiredEffort`, `completedEffort`, and `remainingEffort` calculations.

## API & Routing Fixes
* **Product Editing 405 Error:** `ProductsPanel` sent `PUT` but the API route exported `PATCH`. Changed frontend to `PATCH`.
* **Multi-Date Override Bug:** A global uniqueness check blocked capacity overrides for more than one date. Removed the faulty check.
* **Progress Update No-Auth:** `POST /api/admin/capacity/progress` had no `requireAdmin()` guard. Added it.
* **Accept Validation:** `shippingDurationDays` schema used `.int()`, rejecting valid decimal inputs like `3.5`. Changed to `.number().min(0).max(365)`.
* **Dead Code Cleanup:** Deleted unused `GET /api/capacity/available` endpoint and all dead functions in `lib/capacity.ts`.

## UI/UX & Performance
* **Keystroke Debounce:** Shipping duration input in `OrderCard` now debounces preview requests by 400ms, preventing a DB query storm on every character typed.
* **State Isolation:** Extracted `OrderCard` and `ProductItem` components to localize state, preventing re-renders of entire lists on a single item change.
* **Server-Side Data Fetching:** Migrated initial data fetching for Admin Orders, Products, and Customer Tracking pages to the server, improving FCP and reducing client-side loading flashes.
* **Build-Time DB Exhaustion:** Added guard to skip event tracking during `next build` and marked data-heavy pages as `force-dynamic` to prevent pool exhaustion on free-tier databases.

# Logic Validation Scenarios (Verified)

## Scenario 1: Mid-Day Progress (Locking)
* **State:** Order A needs 2.0 units. Daily limit is 1.0.
* **Action:** Admin records 0.5 units done today.
* **Result:** Scheduler locks those 0.5 units for Order A. Order A's remaining 1.5 units are pushed to future dates if necessary. Today's "Planned" total for Order A is exactly 0.5.

## Scenario 2: Urgent Order (Stealing Remaining Capacity)
* **State:** Order A has 0.5 units locked today. Limit is 1.0. Remaining today = 0.5.
* **Action:** Urgent Order B arrives (needs 0.5, deadline TODAY).
* **Result:** Order B is allocated the remaining 0.5 units today. Order A is pushed to the next available slot. Both meet their deadlines.

## Scenario 3: Capacity Override (Staff Shortage)
* **State:** Limit is 1.0.
* **Action:** Admin overrides limit for tomorrow to 0.2.
* **Result:** Scheduler only allocates 0.2 units across all orders for tomorrow, pushing the overflow to the following days.

## Scenario 4: Constraint Failure
* **State:** Order A has 0.5 locked today. Remaining today = 0.5.
* **Action:** Urgent Order C arrives (needs 0.8, deadline TODAY).
* **Result:** Scheduler rejects Order C with "Missing 0.30 units" reason. System integrity is maintained; Order A is not jeopardized.

# Customer Shared Data and Caching Changes (June 20, 2026)

This section is the authoritative description of the current customer data flow. It supersedes older notes that say every customer page queries Prisma directly, public page renders create guest users, or payment submission lacks ownership verification.

## Shared Server Data Layer

Two server-only data modules now centralize customer-facing reads:

* `src/server/data/products.ts`
  * `getFeaturedProducts()` - cached home-page products, revalidated every 15 minutes.
  * `getDefaultCatalogProducts()` - cached default catalog, revalidated every 15 minutes.
  * `getProductBySlug(slug)` - shared cached product detail, revalidated every hour.
  * All product caches use the `products` cache tag.
* `src/server/data/orders.ts`
  * `getTrackableOrder()` - shared, uncached tracking lookup and Prisma-to-client serialization.
  * `getOrderForPayment()` - uncached order ownership and payment-eligibility lookup.

The project is on Next.js 16.2.9 without Cache Components enabled, so the product data layer uses `unstable_cache`. Server Components and Route Handlers import shared server functions directly; they do not call internal API routes merely to read server data.

## Product Caching and Invalidation

* `/` uses `getFeaturedProducts()` and no longer declares `force-dynamic`.
* `/catalog` remains dynamic because filters come from search parameters. Its default, unfiltered/newest result uses `getDefaultCatalogProducts()`; arbitrary searches, price filters, and sort combinations query Prisma directly to avoid an unbounded cache-key space.
* `/products/[slug]` uses `getProductBySlug()` and no longer declares `force-dynamic`.
* `/order/[slug]` remains `force-dynamic`, but it consumes the same cached `getProductBySlug()` data as the product page.
* Admin product create, update, and deactivate routes call `revalidateTag("products", "max")` after successful mutation.
* `POST /api/orders` deliberately performs a fresh product lookup. Active status and price must be authoritative when money and order records are written, regardless of cached display data.

## Guest Identity and Order Creation

Public discovery pages no longer call `getOrCreateCustomerId()` and no longer write analytics events during Server Component rendering or route prefetching. Browsing home, catalog, product, or order pages therefore does not create a `User` row.

Current order creation flow:

1. `OrderForm` sends product and order input only; it does not send or trust a client-provided `userId`.
2. `POST /api/orders` validates the request and performs a fresh active-product/price lookup.
3. The route reads the signed `bouq***` cookie using `getCurrentUserId()`.
4. If the cookie has no valid database user, the route creates a guest user and sets the signed customer session.
5. The route creates the address, order, order item, and product order-count update.
6. The browser continues to `/login` with only the order ID and order number.

`POST /api/customer/login` no longer accepts a guest user ID from the request body. It derives the guest identity from the signed session, then updates that user or safely merges its orders/addresses into an existing matching customer.

## Tracking

The tracking page and `GET /api/orders/track` now share `getTrackableOrder()` instead of maintaining duplicate Prisma queries and duplicate serialization logic.

Tracking remains dynamic and uncached because order/payment status changes frequently. Access requires:

* an order number plus ownership through the signed customer session, or
* an order number plus the matching mobile number for direct-entry tracking.

An order number alone is no longer sufficient. The UI copy, submit-button validation, page lookup, and API lookup all follow this rule.

## Payment Authorization

The payment page explicitly remains `force-dynamic`. It now accepts only `orderId` as continuation input and derives customer identity from the signed cookie. URL-supplied `userId`, mobile number, and order number are no longer authorization inputs.

Both the payment page and `POST /api/orders/[id]/payment` verify:

* the signed-session user owns the order; and
* the order is in `ACCEPTED`, `PAYMENT_PENDING`, or `PAYMENT_REJECTED` status.

Tracking links to payment now contain only the order ID. The server supplies the authoritative order number and customer mobile number after ownership validation.
