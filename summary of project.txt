# Bouquet Project – Technical Architecture & Business Flow

## Project Objective

A bouquet order management and production planning system that ensures orders are only accepted when sufficient production capacity exists between the order request date and production deadline.

The system combines:

* Product catalog management
* Customer order management
* Payment verification
* Production capacity planning
* Automatic scheduling
* Delivery tracking

The core problem being solved is:

> "Given limited production effort per day, can a new order be completed before its production deadline without causing existing accepted orders to miss theirs?"

---

# Core Concepts

## Daily Capacity

Each date has a configurable production capacity stored in the `Capacity` table (field: `maximumCapacity`).

Default capacity constant: `DAILY_PRODUCTION_CAPACITY = 1` (defined in `lib/constants.ts`).

Per-date overrides are stored in the `Capacity` table and take precedence over the constant.

Example:

| Date   | Capacity |
| ------ | -------- |
| 15 Jun | 1        |
| 16 Jun | 1        |
| 17 Jun | 1        |

Capacity is measured in **production effort days**, not bouquet count.

```text
1   = one full production day
0.5 = half production day
2   = two production days available
```

Capacity can be manually adjusted from CapacityPanel via `POST /api/admin/capacity/override`.

The `Setting` model also stores a `DAILY_PRODUCTION_CAPACITY` key used by `getDailyProductionCapacity()` in `lib/capacity.ts` (legacy helper — the active scheduling engine uses `DAILY_PRODUCTION_CAPACITY` constant from `lib/constants.ts` as fallback).

---

## Product.productionDays

Stored as `Decimal(10,2)` on the `Product` model. Represents effort required to produce one unit of a product.

Example:

| Product         | productionDays |
| --------------- | -------------- |
| Rose Bouquet    | 1              |
| Premium Bouquet | 2              |
| Wedding Bouquet | 3              |

---

## Order Required Capacity

Order effort is calculated from order items at multiple points in the codebase.

Formula:

```text
requiredCapacity = Σ (item.product.productionDays × item.quantity)
```

Used in:
- `api/orders/[id]/accept/route.ts` — before accepting
- `api/admin/capacity/route.ts` — for capacity preview in OrderPanel
- `lib/scheduler.ts → getSchedulerData()` — when building the schedule

---

## Production Deadline

```text
occasionDate
    ↑
shippingDate  = occasionDate - shippingDurationDays
    ↑
productionDeadline = shippingDate - 1 day
```

Calculated in `lib/capacity.ts`:
- `calculateShippingDate(occasionDate, shippingDurationDays)`
- `calculateProductionDeadline(shippingDate)` — subtracts 1 day

Set on the Order when admin accepts it (via `/api/orders/[id]/accept`).

# Order Status Flow

```
PENDING_REVIEW
  → ACCEPTED           (admin accepts via OrderPanel)
  → REJECTED           (admin rejects)

ACCEPTED
  → PAYMENT_PENDING    (after acceptance, customer needs to pay)

PAYMENT_PENDING / PAYMENT_SUBMITTED
  → PAYMENT_VERIFICATION  (customer submits payment screenshot)

PAYMENT_VERIFICATION
  → CONFIRMED          (admin verifies payment)
  → PAYMENT_REJECTED   (admin rejects payment)

CONFIRMED → IN_PRODUCTION → READY_TO_SHIP → SHIPPED → DELIVERED

Any non-terminal → CANCELLED / REFUNDED
```

### SCHEDULABLE_STATUSES (orders included in scheduling):
```
ACCEPTED, PAYMENT_PENDING, PAYMENT_SUBMITTED, PAYMENT_VERIFICATION,
CONFIRMED, IN_PRODUCTION, READY_TO_SHIP
```

### INACTIVE_STATUSES (excluded from scheduling):
```
CANCELLED, REFUNDED, REJECTED, SHIPPED, DELIVERED, PAYMENT_REJECTED
```

Both defined in `lib/constants.ts`.

# Scheduling Engine

Location: `lib/scheduler.ts`

Main exported functions:
- `scheduleOrdersInMemory()`
- `getSchedulerData()`
- `checkAcceptability()`
- `rebuildSchedule()`
- `getSchedulerPlanningRows()`

---

## scheduleOrdersInMemory(todayKey, orders, capacityLimits, completedWorkMap)

Pure in-memory scheduling algorithm. No DB access.

Parameters:
- `todayKey`: YYYY-MM-DD string for today
- `orders`: `OrderScheduleInput[]` — each has `id`, `orderNumber`, `productionDeadlineKey`, `requiredCapacity`, `lockedCapacity`, `status`
- `capacityLimits`: `Map<dateKey, maxCapacity>` — per-date limits from DB
- `completedWorkMap`: `Map<dateKey, completedWork>` — already-done work per date

Algorithm:
1. For each order, compute `remaining = requiredCapacity - lockedCapacity`
2. Build list of viable dates from `todayKey` to `productionDeadlineKey`
3. Compute `slack = totalWindowCapacity - remaining`
4. Sort orders: most constrained first (fewest dates → lowest slack → earliest deadline)
5. Walk forward through dates, allocate `min(need, available)` per day
6. If any order can't be fully allocated before its deadline → return `success: false` with reason

Returns: `{ success, allocations: Map<orderId, ScheduledAllocation[]>, reason? }`

---

## getSchedulerData(todayKey)

Pulls current scheduling context from DB.

Steps:
1. Fetch all orders with `SCHEDULABLE_STATUSES` that have a `productionDeadline`
2. Find the farthest `productionDeadline` among non-inactive orders (determines window end)
3. Fetch `Capacity` records in `[today, endDate]` window → build `capacityLimits` map
4. Fetch all `CapacityReservation` records in the window → build `completedWorkMap` (sum of `completedQuantity` per date)
5. Map orders to `OrderScheduleInput` — `lockedCapacity` = sum of `completedQuantity` from reservations

Returns: `{ inputs, capacityLimits, completedWorkMap }`

---

## checkAcceptability(candidate)

Used before accepting an order to validate capacity.

Parameters: `{ id, orderNumber, productionDeadline: Date, requiredCapacity }`

Process:
1. Calls `getSchedulerData(todayKey)` to get current state
2. Removes the candidate order from inputs (to re-add it fresh)
3. Adds candidate with `lockedCapacity: 0` and `status: "ACCEPTED"`
4. Runs `scheduleOrdersInMemory()` with the updated inputs
5. If success → extracts allocation for candidate → returns `suggestedDates`

Returns:
```typescript
{ canAccept: boolean, reason?: string, suggestedDates: Array<{ date: string, quantity: number }> }
```

---

## rebuildSchedule()

Rebuilds ALL future capacity reservations from scratch.

Process:
1. Calls `getSchedulerData(todayKey)`
2. Runs `scheduleOrdersInMemory()`
3. Upserts `Capacity` rows for every allocation date (outside transaction, required before FK inserts)
4. Builds a `Map<dateKey, capacityId>` from DB
5. In a DB transaction:
   - Deletes all `isManual: false` reservations with `date >= today`
   - Creates new `CapacityReservation` records using flat `capacityId` scalar (not nested connect — `createMany` does not support relations)
6. Calls `getSchedulerPlanningRows()` and returns rows

Throws if `scheduleOrdersInMemory()` returns `success: false`.

**Note:** `rebuildSchedule()` is now only called when a status change actually moves an order into or out of SCHEDULABLE_STATUSES.

---

## getSchedulerPlanningRows()

Builds the planning table for the admin capacity UI.

Process:
1. Finds the farthest `occasionDate` among non-inactive orders (window end)
2. Fetches `Capacity` rows from today to window end, with full reservation joins (order → user → addresses, items → product)
3. Walks each date in the window day-by-day
4. For dates with a `Capacity` record: computes `used` (sum of `plannedQuantity`), `remaining`, `isFull`
5. Skips dates with no `Capacity` record

Returns array of rows (used by `GET /api/admin/capacity` when no query params).

---

# Capacity Reservation Model

```
CapacityReservation
  capacityId        → Capacity (date)
  orderId           → Order
  plannedQuantity   → effort allocated for future work (Decimal)
  completedQuantity → effort already done (Decimal, default 0)
  manualQuantity    → manual override amount (Decimal, default 0)
  isManual          → true for manual overrides (not deleted on rebuild)
```

Unique constraint: `(capacityId, orderId)` — one reservation per order per date.

`rebuildSchedule()` only deletes `isManual: false` future reservations, preserving manual ones.

- Body: `{ orderId, date, completedUnits }`

- Note: validation for `completedUnits > remaining` is commented out

## API: GET /api/admin/orders

Params: `status` (optional)

Returns orders with:
- user (name, mobileNo, email, addresses)
- items (with product.name, product.productionDays)
- payments (latest one: screenshotUrl, status)
- Computed `maxProductionDays` = max of item.product.productionDays (or 1)

---

## API: PATCH /api/admin/orders/[id]

Body schema (zod):
```typescript
{
  status?: OrderStatus,
  trackingNumber?: string,
  verifyPayment?: boolean,
  rejectPayment?: boolean
}
```

Logic:
1. If `verifyPayment: true`:
   - Sets all PENDING payments → VERIFIED (with verifiedAt)
   - Sets order status → CONFIRMED
   - **No rebuildSchedule()** — PAYMENT_VERIFICATION and CONFIRMED are both in SCHEDULABLE_STATUSES, schedule is unchanged

2. If `rejectPayment: true`:
   - Sets all PENDING payments → REJECTED
   - Sets order status → PAYMENT_REJECTED
   - Calls `rebuildSchedule()` — PAYMENT_REJECTED is INACTIVE, order leaves the schedule

3. Generic status / trackingNumber update:
   - Updates order in DB
   - Calls `rebuildSchedule()` only if `schedulingAffected(fromStatus, toStatus)` is true — i.e. the transition crosses the SCHEDULABLE ↔ INACTIVE boundary
   - CONFIRMED → IN_PRODUCTION → READY_TO_SHIP: no rebuild (all schedulable)
   - SHIPPED, DELIVERED, CANCELLED, REFUNDED: rebuild (leaves schedulable set)
   - Tracking-number-only update: no rebuild

---

## API: POST /api/orders/[id]/accept

Called by OrderPanel's "Accept Order" button.

Process:
1. Requires admin auth
2. Parses `{ shippingDurationDays }` from body
3. Fetches order with items + productionDays
4. Validates `occasionDate` exists
5. Computes `shippingDate` and `productionDeadline`
6. Computes `requiredCapacity = Σ(item.quantity × productionDays)`
7. Calls `checkAcceptability(...)` ← **second call (first was from OrderPanel preview)**
8. If `!check.canAccept` → returns 409 with reason
9. Updates order:
   - status → ACCEPTED
   - shippingDurationDays, shippingDate, productionDeadline, occasionDate
   - Updates all payments to status PENDING (resets any prior payment state)
10. Calls `rebuildSchedule()`
11. Returns `{ success: true, suggestedDates }`

# Fixes Applied (June 2026)

## Fix 1 — CRITICAL: rebuildSchedule() createMany used nested relation connect (broke accept)

**Root cause:** `rebuildSchedule()` in `lib/scheduler.ts` was building reservation records like:
```typescript
creates.push({
  orderId,
  capacity: { connect: { date } },  // ← Prisma createMany does NOT support nested relations
  ...
});
await tx.capacityReservation.createMany({ data: creates });
```
Prisma's `createMany` only accepts flat scalar fields. The nested `connect` was silently stripped, causing the insert to fail with a constraint violation (missing `capacityId`). This broke every call to `rebuildSchedule()`, which is called at the end of the accept flow — making order acceptance fail.

**Fix:** Before the transaction, upsert `Capacity` rows for all allocation dates, then build a `Map<dateKey, capacityId>` lookup. The `createMany` now uses flat `capacityId` scalar directly:
```typescript
creates.push({ capacityId, orderId, plannedQuantity: a.quantity, ... });
```

Also fixed: the previous code only ensured Capacity rows for dates already in `capacityLimits` (fetched from DB). New orders with deadlines beyond the current DB window had no Capacity row to connect to. Now all allocation dates are upserted before the transaction.

Files changed: `src/lib/scheduler.ts`

---

## Fix 2 — rebuildSchedule() called unconditionally on every status change

**Root cause:** `PATCH /api/admin/orders/[id]` called `rebuildSchedule()` on every `body.status` change regardless of whether the change affected the production schedule.

Unnecessary rebuilds were triggered for:
- `verifyPayment` → CONFIRMED (both PAYMENT_VERIFICATION and CONFIRMED are in SCHEDULABLE_STATUSES — no change to schedule)
- CONFIRMED → IN_PRODUCTION (both schedulable — no change)
- IN_PRODUCTION → READY_TO_SHIP (both schedulable — no change)

**Fix:** Added a `schedulingAffected(fromStatus, toStatus)` helper that checks whether the transition crosses the boundary between SCHEDULABLE_STATUSES and INACTIVE_STATUSES. `rebuildSchedule()` is only called when this returns true.

Rebuild now fires only for:
- PAYMENT_REJECTED (leaves schedulable set) ✓
- SHIPPED, DELIVERED, CANCELLED, REFUNDED (leave schedulable set) ✓

Does NOT rebuild for:
- `verifyPayment` → CONFIRMED ✗ (removed — same schedulable set)
- CONFIRMED → IN_PRODUCTION ✗
- IN_PRODUCTION → READY_TO_SHIP ✗
- Tracking number only update ✗

Files changed: `src/app/api/admin/orders/[id]/route.ts`

---

## Fix 3 — Capacity override blocked after first date (multi-date override bug)

**Root cause:** `POST /api/admin/capacity/override` contained a global uniqueness check:
```typescript
const existingGlobalManual = await prisma.capacityReservation.findFirst({ where: { isManual: true } });
if (existingGlobalManual && existingGlobalManual.capacityId !== capacity.id) {
  return 409 "Only one manual capacity override can exist at a time"
}
```
This prevented overriding more than one date. The second date override always failed with 409.

**Fix:** Removed both the global uniqueness check and the redundant per-date check. The override now simply upserts `maximumCapacity` on the `Capacity` record and logs to `AuditLog`. The `override-system` CapacityReservation approach was also removed — `maximumCapacity` on the `Capacity` record is the single source of truth for daily limits.

Files changed: `src/app/api/admin/capacity/override/route.ts`

---

## Fix 5 — Product editing broken (PUT vs PATCH mismatch)

`ProductsPanel.tsx` sent `method: "PUT"` but the API route only exports `PATCH`. Every edit product request returned 405 Method Not Allowed.

Fixed: changed `method: "PUT"` → `method: "PATCH"` in `ProductsPanel.tsx`.

Files changed: `src/app/admin/products/ProductsPanel.tsx`

---

## Fix 6 — verifyPayment never actually updated the Payment record

The PATCH route filtered payments `where: { status: "PENDING" }` but `POST /api/orders/[id]/payment` sets payment status to `"UPLOADED"`, not `"PENDING"`. The filter matched zero rows so `verifiedAt` was never set and the payment record stayed in UPLOADED state indefinitely.

Fixed: changed filter to `where: { status: "UPLOADED" }`.

Files changed: `src/app/api/admin/orders/[id]/route.ts`

---

## Fix 7 — POST /api/admin/capacity/progress had no auth guard

The progress update endpoint was callable by anyone without authentication. `requireAdmin()` was missing entirely.

Fixed: added auth guard at the top of the handler. Removed the unused `getRemainingCapacity` import and replaced with `requireAdmin` from `lib/auth`.

Files changed: `src/app/api/admin/capacity/progress/route.ts`

---

## Fix 8 — DB connection string logged to stdout on every cold start

`lib/prisma.ts` had two `console.log` calls that printed the database host parsed from `DATABASE_URL` and `DIRECT_URL`. This leaks connection info to any log aggregator in production.

Fixed: removed both console.log calls.

Files changed: `src/lib/prisma.ts`

# Changes Applied (Round 3 — June 2026)

## Fix 9 — Accept order button: shippingDurationDays .int() validation rejected valid inputs

**Root cause:** `acceptSchema` had `z.number().int()`. The admin input field can naturally produce a non-integer (e.g. entering `3.5` for 3.5 days shipping). Zod's `.int()` would reject it and the `safeParse` failure returned "Shipping duration is required" — the actual reason was hidden.

**Fix:** Changed to `z.number().min(0).max(365)` — integers and decimals both accepted.

Files changed: `src/app/api/orders/[id]/accept/route.ts`

---

## Fix 10 — Progress update: timezone mismatch caused findFirst to return nothing

**Root cause:** The progress route used `toDateOnly()` from `lib/utils.ts` which calls `setHours(0,0,0,0)` — local time. But `Capacity.date` is stored as UTC midnight. In any timezone with a positive UTC offset, local midnight is a different UTC value than the stored date, so `prisma.capacityReservation.findFirst({ where: { capacity: { date: dateOnly } } })` matched zero rows every time. The API returned 404 silently.

**Fix:** Rewrote the route to use `formatDateKey(new Date(date), true)` + `parseDateKey()` from `lib/scheduler.ts` — both are pure UTC and match how `Capacity.date` is stored. Additional improvements:
- Rejects `completedUnits <= 0`
- Caps `completedQuantity` at `plannedQuantity` (can't mark more work done than was allocated for that date)
- Returns `{ completedQuantity }` so the UI can update optimistically

Files changed: `src/app/api/admin/capacity/progress/route.ts`

## Fix 11 — CapacityPanel expanded view showed wrong data in all columns

**Root cause:** All four columns (Production Days, Allocated, Completed, Remaining) were wired to `order.bouquetQuantity` which was `res.plannedQuantity` — the effort slice for this specific date only. This made all columns show the same number.

The correct values require:
- `totalRequiredEffort` = `Σ(product.productionDays × item.quantity)` across all items — the total production cost of the order
- `allocatedToday` = `plannedQuantity` on this specific date — what's scheduled for today
- `completedEffort` = sum of `completedQuantity` across ALL reservations for this order (all dates)
- `remainingEffort` = `totalRequiredEffort - completedEffort`

**Fix:** Three-part change:

**`lib/scheduler.ts` — `getSchedulerPlanningRows()`**: Added `capacityReservations: true` to the order include so all per-date completedQuantity values are available when computing order-level totals.

**`src/app/api/admin/capacity/route.ts`**: Replaced `bouquetQuantity: res.quantity` with four explicit fields per order:
```typescript
allocatedToday: res.quantity,           // effort for this date
totalRequiredEffort,                     // Σ(productionDays × qty)
completedEffort,                         // Σ(completedQuantity) all dates
remainingEffort: max(0, total - completed)
```
Also filters out `isManual: true` reservations from the orders list (the `override-system` entry was appearing as a row).

**`src/app/admin/capacity/CapacityPanel.tsx`**: Full rewrite of the expanded table:
- Updated `PlanningOrder` type with the four new fields
- Columns: Order, Customer, Items (product name × qty × productionDays), Total Effort, Allocated Today, Completed, Remaining, Deadline, Status, Mark Work Done
- `Mark Work Done` input is capped at `allocatedToday` (max you can mark is what was planned for today)
- On success, updates display optimistically from the API response without a full reload
- Shows `alert()` on error instead of silently logging to console

Files changed: `src/lib/scheduler.ts`, `src/app/api/admin/capacity/route.ts`, `src/app/admin/capacity/CapacityPanel.tsx`

# Remaining Known Issues (not yet fixed)

## 1. Customer cookie stores raw userId (unsigned)

The `bouquet_customer` cookie holds the plain Prisma user ID. Any client who can forge this cookie gains access as that user. Low risk for a private tool.

## 2. No ownership check on /api/orders/[id]/payment

Any caller who knows an order ID can submit a payment screenshot for it. Should cross-check against session user.

## 3. ProductionProgress model unused

The `ProductionProgress` DB model exists in the schema but nothing reads or writes it. Safe to drop with a migration.

## 4. getDailyProductionCapacity vs DAILY_PRODUCTION_CAPACITY constant

`lib/scheduler.ts` uses `DAILY_PRODUCTION_CAPACITY = 1` as fallback. The `Setting` table value has no effect on the scheduler unless a `Capacity` row exists for that date.

## 5. Request-Date Boundary Not Enforced

The scheduler allocates from `todayKey` forward but doesn't store the original order request date.

## 6. Double checkAcceptability Call (acceptable, low priority)

OrderPanel preview calls `checkAcceptability` on every shipping duration change. The accept API calls it again authoritatively. The API call is the gate — the preview call is cosmetic.

## One-Line Project Description

**Bouquet is a capacity-aware bouquet production planning and order management platform that calculates production effort from product production days, validates capacity before order acceptance via an in-memory scheduling algorithm, allocates fractional daily effort across production dates using CapacityReservation records, and rebuilds future schedules whenever accepted orders change to ensure all schedulable orders meet their production deadlines.**
